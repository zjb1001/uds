# UDS 诊断系统 - 设计文档总览

## 项目概述

本项目目标是建立一套完整的 **UDS 诊断模拟刷写系统**，符合汽车电子 UDS 标准（ISO 14229 系列），利用 Linux 虚拟 CAN 通信实现诊断功能和 ECU 固件刷写模拟。

### 核心目标

1. **UDS 诊断服务实现** - 实现 ISO 14229-1 标准中的核心诊断服务
2. **ECU 固件刷写模拟** - 模拟 ECU 刷写流程（解锁→擦除→编程→验证）
3. **虚拟 CAN 通信** - 基于 Linux SocketCAN 实现多 ECU 虚拟通信
4. **完整工具链** - 提供诊断仪、刷写工具、测试框架

### 关键设计原则

1. **模块化设计** - 各层相互独立，通过接口交互
2. **标准兼容** - 严格遵循 ISO 14229-1 标准规范
3. **高可扩展性** - 支持新诊断服务、新 ECU 类型的灵活添加
4. **高可靠性** - 异常处理、超时管理、数据校验完善
5. **可测试性** - 完整的单元测试、集成测试、验收测试框架

## 项目关键问题与决策

### 1. UDS 服务覆盖范围

**决策**: 实现 ISO 14229-1 核心诊断服务（共 22 个）的关键子集

**支持服务列表**:
- `0x10` - Diagnostic Session Control (诊断会话控制)
- `0x11` - ECU Reset
- `0x14` - Clear Diagnostic Information
- `0x19` - Read DTC Information
- `0x22` - Read Data By Identifier
- `0x24` - Read Scaling Data Information
- `0x26` - Read Data By Address and Length
- `0x27` - Security Access (解锁机制)
- `0x2E` - Write Data By Identifier
- `0x31` - Routine Control (诊断程序)
- `0x34` - Request Download (刷写前置)
- `0x35` - Request Upload (读取)
- `0x36` - Transfer Data (数据传输)
- `0x37` - Request Transfer Exit (传输结束)
- `0x3D` - Write Memory by Address
- `0x3E` - Tester Present (心跳检测)
- `0x85` - Disable Normal Reception / Transmission

### 2. ECU 模拟复杂度

**决策**: 单 ECU 完整模拟 + 多 ECU 扩展支持

**初期范围**:
- 单个虚拟 ECU，完整诊断会话支持
- 内存模型（RAM）、闪存模型（Flash）模拟
- DTC（故障代码）记录和管理
- 诊断数据流（Data Identifier）支持

**扩展范围**:
- 多 ECU 网络总线模拟
- ECU 之间的报文转发和 Gateway 功能

### 3. 安全与加密

**决策**: 支持基础安全访问，预留高级加密扩展

**初期**: 
- `0x27` 安全访问（简化密钥验证）
- 会话级权限管理（编程会话 vs 标准会话）

**扩展**:
- 密钥生成算法（如 XOR、SHA）
- SecOC（安全车载通信）

### 4. 刷写数据来源

**决策**: 支持文件读取和网络下载

**格式支持**:
- Motorola S-Record (.srec, .s37)
- Intel HEX (.hex)
- Bin 二进制格式
- 内部模拟数据生成

### 5. 跨平台支持

**决策**: 优先 Linux，后续扩展至 Windows

**初期**: Linux SocketCAN 原生支持
**扩展**: Docker 容器化 + Windows WSL2 兼容

## 假设与非目标

- **环境假设**: Linux 主机具备 SocketCAN 与 vCAN 支持, 允许加载 `vcan` 模块; 具备 Rust 1.75+ 与 Python 3.10+ 开发环境。
- **部署假设**: 单机仿真为主, 不要求实时硬件接入; 如需真实 CAN 硬件, 另行评估波特率和时序。
- **非目标 (当前版本)**:
	- 不提供 DoIP、以太网诊断适配。
	- 不实现 OEM 私有 UDS 变体; 仅提供标准子集及示例扩展点。
	- 不做功能安全 (ISO 26262) 证明, 仅做工程级防护与日志。

## 质量与性能基线

- **性能指标对齐**: 响应延迟、刷写速率与并发会话目标与 [01-System_Requirements_Specification.md](01-System_Requirements_Specification.md#L90-L140) 保持一致; 若实现偏差需在发布说明中注明。
- **可靠性**: 关键路径 (会话切换、刷写) 均需具备超时、重试与幂等保护; 负响应需覆盖常见 NRC。
- **可观测性**: 诊断请求/响应、会话切换、刷写阶段必须产生日志 (INFO) 与异常日志 (WARN/ERROR), 以支持测试与问题定位。
- **扩展性**: 新增 ECU 或服务应通过配置/插件化方式完成, 不修改核心协议栈。
