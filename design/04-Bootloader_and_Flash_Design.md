## 2. Bootloader 概述

### 2.1 Bootloader 功能定义

**核心职责**:
1. **上电初始化**: 检查应用程序有效性,决定启动 Bootloader 或 Application
2. **刷写管理**: 通过 UDS 协议接收和编程固件
3. **完整性验证**: 验证刷写数据的完整性和有效性
4. **安全保护**: 防止非法刷写和未授权访问

### 2.2 Bootloader 工作模式

```
┌─────────────────────────────────────────────────────────┐
│                    Bootloader                           │
│  ┌──────────────┬──────────────┬──────────────┐         │
│  │ 正常启动模式 │ 刷写模式     │ 恢复模式     │         │
│  └──────────────┴──────────────┴──────────────┘         │
└─────────────────────────────────────────────────────────┘
```

| 模式 | 触发条件 | 行为 |
|------|----------|------|
| **正常启动** | 应用程序有效标志 = 0xAA55 | 跳转到应用程序 |
| **刷写模式** | 收到 0x10 编程会话请求 | 进入刷写流程 |
| **恢复模式** | 应用程序无效或超时 | 等待刷写恢复 |

### 2.3 Bootloader 状态机

```
           系统上电
              ↓
        ┌─────────────┐
        │  CHECK_APP  │ ← 检查应用程序有效性
        └──────┬──────┘
               │
        ┌──────┴──────┐
        │             │
   App 有效      App 无效
        │             │
        ↓             ↓
  ┌──────────┐  ┌──────────┐
  │ JUMP_APP │  │ BOOT_MODE│
  └──────────┘  └─────┬────┘
                      │
           收到编程请求 (0x10)
                      ↓
              ┌──────────────┐
              │ FLASH_MODE   │
              │  - 解锁      │
              │  - 擦除      │
              │  - 编程      │
              │  - 验证      │
              └──────┬───────┘
                     │
           刷写完成 (0x37)
                     ↓
              ┌──────────────┐
              │ VERIFY_APP   │
              └──────┬───────┘
                     │
              ┌──────┴──────┐
              │             │
         验证成功      验证失败
              │             │
              ↓             ↓
        ┌──────────┐  ┌──────────┐
        │ RESET    │  │ ERROR    │
        └──────────┘  └──────────┘
```

**状态标志来源**: 上述跳转决策依赖刷写元数据区的 `app_valid`、`update_pending` 和 `active_bank` 标志,确保断电后仍可判定是否进入恢复模式或跳转应用。

## 3. Flash 存储器模型

### 3.1 Flash 内存布局

**模拟内存大小**: 1 MB (默认,可配置,最小 512 KB)

```
地址范围           大小       说明
─────────────────────────────────────────
0x000000 - 0x007FFF   32 KB    Bootloader 区域 (写保护)
0x008000 - 0x00FFFF   32 KB    Bootloader 备份区 (写保护)
0x010000 - 0x0DFFFF   832 KB   应用程序主区 (可刷写,对齐 4 KB)
0x0E0000 - 0x0EFFFF   64 KB    应用程序备份区 (可刷写,对齐 4 KB)
0x0F0000 - 0x0F7FFF   32 KB    参数配置区
0x0F8000 - 0x0FFFFF   32 KB    诊断/刷写元数据区 (DID, DTC, 刷写标志)
```

### 3.2 Flash 分区属性

| 分区 | 起始地址 | 大小 | 访问权限 | 说明 |
|------|----------|------|----------|------|
| Bootloader | 0x000000 | 32 KB | 只读 | Bootloader 代码 |
| Bootloader Backup | 0x008000 | 32 KB | 只读 | Bootloader 备份 |
| Application | 0x010000 | 832 KB | 读/写 | 应用程序 |
| App Backup | 0x0E0000 | 64 KB | 读/写 | 应用程序备份 |
| Parameter | 0x0F0000 | 32 KB | 读/写 | ECU 参数 |
| Diagnostics | 0x0F8000 | 32 KB | 读/写 | 诊断数据 |

### 3.3 Flash 扇区划分

**模拟擦除粒度**:
- **扇区大小**: 4 KB (4096 字节)
- **块大小**: 64 KB (16 个扇区)
- **全片擦除**: 支持整片擦除命令

```
扇区 0:    0x000000 - 0x000FFF  (Bootloader)
扇区 1:    0x001000 - 0x001FFF  (Bootloader)
...
扇区 7:    0x007000 - 0x007FFF  (Bootloader)
扇区 8-207: 0x008000 - 0x0DFFFF (Application)
...
扇区 224-231: 0x0E0000 - 0x0EFFFF (App Backup)
扇区 240-247: 0x0F0000 - 0x0F7FFF (Parameter)
扇区 248-255: 0x0F8000 - 0x0FFFFF (Diagnostics)
```

### 3.4 刷写元数据与标志

**用途**: 记录刷写进度、活动 Bank、应用有效性,支持断点续刷和回滚。

**推荐位置**: 诊断/刷写元数据区的末尾 1 个扇区 (0x0FF000 - 0x0FFFF, 4 KB)。

**数据结构**:
```c
// 对齐 4 字节,整扇区擦写
struct BootFlags {
    uint32_t magic;            // 0xABCD1234 表示结构有效
    uint8_t  active_bank;      // 0=主区(App),1=备份区(App Backup)
    uint8_t  app_valid;        // 0/1, 应用有效标志
    uint8_t  update_pending;   // 0/1, 正在刷写标志
    uint8_t  reserved0;
    uint32_t write_offset;     // 已接收/写入的偏移,支持断点续传
    uint32_t expected_size;    // 目标固件总大小
    uint32_t firmware_crc32;   // 全镜像 CRC32
    uint32_t app_version;      // 版本号,便于回滚比较
    uint32_t header_addr;      // 镜像头部地址 (对齐 16 字节)
    uint32_t entry_point;      // 应用入口地址
    uint32_t reserved[5];
};
```

**更新规则 (原子性保护)**:
- 写入刷写数据前: `update_pending=1`, `app_valid=0`。
- 每块成功写入后: 更新 `write_offset`。
- 刷写完成且校验通过: 写入 `firmware_crc32`、`app_version`、`entry_point` 后,再将 `app_valid=1`, `update_pending=0`。
- 断电恢复: `update_pending=1` 则进入恢复模式,尝试续写或擦除并重刷。

## 4. 固件刷写协议

### 4.0 UDS服务调用映射表

本章涉及的所有UDS服务详细定义见 [03-UDS_Protocol_Design.md](03-UDS_Protocol_Design.md)。下表展示刷写流程中各服务的调用次序与约束：

| 步骤 | 服务 ID | 服务名称 | 必/可选 | 详细参考 | 关键约束 |
|------|---------|---------|--------|---------|---------|
| 前处理-1 | 0x10 | Diagnostic Session Control | 必需 | [03#2.1](03-UDS_Protocol_Design.md#L2.1) | 进入编程会话 (0x10) |
| 前处理-2 | 0x27 | Security Access | 必需 | [03#2.2](03-UDS_Protocol_Design.md#L2.2) | 请求种子(0x01)→计算密钥→发送密钥(0x02)，Level 3 |
| 前处理-3 | 0x28 | Communication Control | 可选 | [03#后续补充](03-UDS_Protocol_Design.md) | 禁用RX/TX避免干扰(可选) |
| 前处理-4 | 0x85 | Control DTC Setting | 可选 | [03#后续补充](03-UDS_Protocol_Design.md) | 禁用DTC记录(可选) |
| 擦除-1 | 0x31 | Routine Control | 可选 | [03#后续补充](03-UDS_Protocol_Design.md) | 执行擦除例程(可选，或由0x34自动擦除) |
| 传输-1 | 0x34 | Request Download | 必需 | [03#后续补充](03-UDS_Protocol_Design.md) | 指定地址/大小/格式，返回maxBlockSize |
| 传输-2 | 0x36 | Transfer Data | 必需 | [03#后续补充](03-UDS_Protocol_Design.md) | 循环发送数据块(blockNumber 0x01-0xFF) |
| 传输-3 | 0x37 | Request Transfer Exit | 必需 | [03#后续补充](03-UDS_Protocol_Design.md) | 终止传输，触发校验 |
| 传输-4(可选) | 0x35 | Request Upload | 可选 | [见本章4.3](#43-请求上传) | 读回已编程数据进行验证 |
| 验证 | 0x31 | Routine Control | 可选 | [03#后续补充](03-UDS_Protocol_Design.md) | 执行校验例程(校验和/签名) |
| 写参数(可选) | 0x3D | Write Memory by Address | 可选 | [见本章4.4](#44-按地址写入内存) | 写入参数/标定数据 |
| 后处理-1 | 0x28 | Communication Control | 可选 | [03#后续补充](03-UDS_Protocol_Design.md) | 恢复RX/TX |
| 后处理-2 | 0x11 | ECU Reset | 必需 | [03#2.0](03-UDS_Protocol_Design.md#L2.0) | 重启ECU，返回Default Session |

**关键规则**:
- 前置条件: 所有操作前需进入编程会话(0x10)+解锁(0x27 Level 3)
- blockNumber连续性: 0x36的blockNumber必须按0x01→0x02→...→0xFF循环
- 原子性: 0x37前发送的所有数据块应在0x34指定的地址范围内无重复
- 超时遵循: 各服务的P2/P2*超时参见[01-System_Requirements_Specification.md](01-System_Requirements_Specification.md#L90-L140)
- 失败回收: 任何步骤失败返回NRC后，重新执行流程需重新进行0x34(新会话)或续传(若块号未完成)

### 4.1 刷写流程总览

```
┌─────────────────────────────────────────────────────────┐
│                    刷写流程                              │
├─────────────────────────────────────────────────────────┤
│  1. 前处理 (Pre-Programming)                             │
│     - 进入编程会话 (0x10)                                │
│     - 安全解锁 (0x27)                                    │
│     - 禁用正常通信 (0x28/0x85)                           │
│                                                          │
│  2. 擦除准备 (Erase Preparation)                         │
│     - 检查内存范围                                       │
│     - 擦除 Flash 区域 (可选)                             │
│                                                          │
│  3. 数据传输 (Data Transfer)                             │
│     - 请求下载 (0x34)                                    │
│     - 传输数据 (0x36)                                    │
│     - 请求传输退出 (0x37)                                │
│                                                          │
│  4. 验证 (Verification)                                  │
│     - 校验和验证                                         │
│     - 签名验证 (可选)                                    │
│     - 完整性检查                                         │
│                                                          │
│  5. 后处理 (Post-Programming)                            │
│     - 使能正常通信                                       │
│     - ECU 重启 (0x11)                                    │
└─────────────────────────────────────────────────────────┘
```

### 4.2 阶段详解

#### 阶段 1: 前处理 (Pre-Programming)

**服务序列**:
```
1. 0x10 0x02 (进入编程会话)
   → ECU: 0x50 0x02 [P2_Server] [P2*_Server]

2. 0x27 0x01 (请求种子 - Level 3)
   → ECU: 0x67 0x01 [Seed]

3. 0x27 0x02 (发送密钥 - Level 3)
   → ECU: 0x67 0x02 (成功) 或 0x7F 27 35 (失败)

4. 0x28 0x03 0x01 (禁用 RX 和 TX, 仅应用)
   → ECU: 0x68 0x03 0x01
```

**错误处理**:
- 0x7F 27 35: 密钥错误,需要重新请求种子
- 0x7F 10 0x22: 会话类型不支持
- 0x7F 28 0x31: 请求超出范围

---

#### 阶段 2: 擦除准备 (Erase)

**关键设计规则 (必读)**:

**版本对比与降级控制**:
```
新版本 < 现有版本:
  ├─ 默认拒绝 (防止意外降级)，返回 NRC 0x22 (conditionsNotCorrect)
  └─ 若需要强制降级: 需特殊配置标志或需要 OEM 权限
  
新版本 ≥ 现有版本:
  └─ 允许升级，正常刷写流程
  
版本比对位置:
  └─ 镜像头部 Version 字段 vs BootFlags.app_version
```

**目标区域选择规则**:
```
情况1: Normal Upgrade (新版本 > 旧版本)
  ├─ 优先选择备用区 (App Backup, 0x0E0000)
  ├─ 成功刷写后，在 0x37 返回时标记 active_bank=1
  ├─ 保留原主区 (Application) 作为回滚点
  └─ 下次启动前验证新版本，若失败则自动回滚

情况2: Recovery (应用无效，需强制覆盖)
  ├─ 直接写入主区 (Application, 0x010000)
  ├─ 不使用备用区
  └─ update_pending=1，断电后进入恢复模式

情况3: Force Update (OEM维护操作)
  ├─ 需要特殊权限标志
  ├─ 可覆盖主区
  ├─ 可禁用版本检查
  └─ 需要日志记录所有覆盖操作

并发防护:
  ├─ 刷写进行中 (update_pending=1) 时禁止重复 0x34
  ├─ 返回 NRC 0x24 (requestSequenceError)
  └─ 需要重启 ECU 或等待前次操作超时清理
```

**断电恢复策略**:
```
上电检查流程:
  ├─ 检查 magic 与 update_pending 标志
  │
  ├─ update_pending=1:
  │  ├─ 检查 active_bank（准备使用的分区）
  │  ├─ 对分区进行 CRC 校验
  │  ├─ 若新分区 CRC 失败 → 清除新数据，回滚使用旧分区
  │  ├─ 若新分区 CRC 通过 → 切换启动，标记 active_bank，清除 update_pending
  │  └─ 若无有效分区 → 进入恢复模式，等待刷写
  │
  └─ update_pending=0:
     └─ 正常启动，按 active_bank 加载应用

恢复模式:
  ├─ 诊断仪可重新发起 0x34/0x36/0x37 流程
  ├─ 支持从断点续传（write_offset 记录）
  ├─ 多次中断后需完全重刷（清除 write_offset）
  └─ 恢复失败次数过多 (>3) 时禁用自动回滚，保留最后一次状态供分析
```

---

#### 阶段 2.1: 擦除方式选择

**擦除方式**:
- **方式1: 自动擦除** - Request Download 时自动擦除目标区域
- **方式2: 显式擦除** - 通过例程控制 (0x31) 擦除指定扇区

**写入目标选择 (防回退失败)**:
- 正常升级: 写入备用区 (App Backup),成功后切换 `active_bank=1`,保留原主区可回滚。
- 恢复/强制覆盖: 写入主区 (Application),跳过备份。
- 规则: 仅允许在 `update_pending=0` 时切换目标区,防止并发刷写。

**例程擦除** (0x31):
```
请求: 0x31 0x01 0xFF 0x01 [Addr_H] [Addr_M] [Addr_L] [Size_H] [Size_L]
      └─ 启动例程 ┘     └─ 擦除内存例程 ─┘ └── 地址 ─┘ └── 大小 ─┘

响应: 0x71 0x01 0xFF 0x01 [执行状态]
```

**擦除时间估算**:
- 单个扇区 (4 KB): ≤ 50 ms
- 整片擦除 (1 MB): ≤ 10 s (模拟环境可忽略)

#### 阶段 3: 数据传输 (Data Transfer)

##### 3.1 请求下载 (0x34 - Request Download)

**请求格式**:
```
┌──────────────┬─────────────┬─────────────┬─────────────┬─────────────┬──────────────┐
│ SID (0x34)   │ dataFormat  │ addressAnd  │ addressAnd  │ addressAnd  │ memorySize  │
│ 1 Byte       │ Identifier  │ LengthFormat│ LengthFormat│ LengthFormat│              │
│              │ 1 Byte      │ 1 Byte      │ 1-4 Bytes   │ 1-4 Bytes   │ 1-4 Bytes    │
└──────────────┴─────────────┴─────────────┴─────────────┴─────────────┴──────────────┘

dataFormatIdentifier:
  - Bit 7-4: 数据压缩方式 (0 = 不压缩)
  - Bit 3-0: 数据加密方式 (0 = 不加密)

addressAndLengthFormatIdentifier:
  - Bit 7-4: 地址字节数 (0x4 = 4 字节)
  - Bit 3-0: 大小字节数 (0x4 = 4 字节)
```

**示例**:
```
请求: 0x34 0x00 0x44 0x00 0x01 0x00 0x00 0x00 0x80 0x00 0x00
    └─ 下载 ─┘ └─ 格式 ─┘ └──── 地址 0x00010000 ───┘ └─ 大小 0x8000 (32KB) ─┘

响应: 0x74 0x00 0x44 [MaxLength_Low] [MaxLength_High]
      └─ 最大块大小 (示例: 0xFF 0x03 = 1023 字节)
```

**处理逻辑**:
```
收到 0x34 请求
    ↓
验证会话类型 (必须是编程会话)
    ↓
验证安全级别 (Level 3)
    ↓
解析地址和大小
    ↓
检查地址范围 (不能是 Bootloader 区)
    ↓
计算可接受的块大小 (根据 Flash 页大小)
    ↓
返回 0x74 + maxBlockSize
```

##### 3.2 传输数据 (0x36 - Transfer Data)

**关键约束规则**:

**块序列号连续性与幂等性**:
```
期望序列: 0x01 → 0x02 → 0x03 → ... → 0xFF → 0x01 (循环)

场景1: 正常顺序
  请求: 0x36 0x01, 0x36 0x02, 0x36 0x03
  响应: 0x76 0x01, 0x76 0x02, 0x76 0x03 ✓

场景2: 重复相同 blockNumber (幂等性)
  请求: 0x36 0x02, 0x36 0x02 (重复)
  处理规则:
    ├─ 若数据完全相同 → 返回之前的响应 0x76 0x02 ✓
    ├─ 若数据不同 → 返回 NRC 0x24 (requestSequenceError)
    └─ 记录重复并计数，超过3次则终止会话

场景3: 乱序接收
  请求: 0x36 0x01, 0x36 0x03 (跳过0x02)
  处理: 返回 NRC 0x24 (requestSequenceError)，重新发送从0x02开始

场景4: 循环归零
  请求: ... 0x36 0xFF, 0x36 0x01 (循环)
  处理: ✓ 允许循环，用于超过255个块的大文件
```

**超时与流量控制参数**:
```
参数 | 值 | 说明 | 来源 |
-----|-----|------|------|
P2 | 50ms | 单帧(0x36)响应超时 | [01-SRS#L100-L110](01-System_Requirements_Specification.md#L100) |
P2* | 2000ms | 多帧传输流控超时 | [01-SRS#L100-L110](01-System_Requirements_Specification.md#L100) |
STmin | 5ms | 连续帧间隔 | [03-UDS#L180-L200](03-UDS_Protocol_Design.md#L180) |
BS | 4 | 流控块大小(每4帧请求FC) | 可配置 |
N_Bs | 1000ms | 等待FC帧超时 | ISO 15765-2 标准 |
N_Cr | 1000ms | 等待CF帧超时 | ISO 15765-2 标准 |

违反规则后果:
  └─ 若某块超过 P2* 仍未接收到响应，诊断仪可发起会话重置
  └─ ECU 应清理未完成的刷写状态，准备新的 0x34 请求
```

**请求格式**:
```
┌──────────────┬──────────────┬──────────────────┐
│ SID (0x36)   │ blockNumber  │ transferData     │
│ 1 Byte       │ 1 Byte       │ variable         │
└──────────────┴──────────────┴──────────────────┘

blockNumber:
  - 0x01: 第一个数据块
  - 0x02-0xFF: 后续数据块 (循环使用)
  - 0x00: 预留 (不使用)

transferData:
  - 长度 ≤ RequestDownload 时返回的 maxBlockSize
  - 包含实际的固件数据
```

**示例**:
```
第一个数据块:
0x36 0x01 [Data... up to 1023 bytes]

响应: 0x76 0x01 (确认)

后续数据块:
0x36 0x02 [Data...]
0x36 0x03 [Data...]
...

最后一个数据块:
0x36 0xFF [Data...]  (最后一块,长度可能 < maxBlockSize)
```

**流量控制**:
```
发送端 (Tester)        接收端 (ECU)
     │                      │
     ├─ 发送 FF (首帧) ───→ │
     │                      ├─ 返回 FC (流控)
     │ ←─────────────────────┤
     │   FC: BS=4, STmin=5ms │
     │                      │
     ├─ 发送 CF (连续帧) ─→ │
     ├─ 发送 CF ─────────→  │
     ├─ 发送 CF ─────────→  │
     ├─ 发送 CF ─────────→  │
     │                      │
     │ ← 发送下一块 ─────────┤

BS (BlockSize): 连续发送多少帧后等待
STmin (SeparationTime): 帧间最小间隔 (毫秒)
```

##### 3.3 请求传输退出 (0x37 - Request Transfer Exit)

**请求格式**:
```
┌──────────────┐
│ SID (0x37)   │
│ 1 Byte       │
└──────────────┘
```

**响应格式**:
```
┌──────────────┬──────────────┐
│ SID (0x77)   │ transferResp │
│ 1 Byte       │ variable     │
└──────────────┴──────────────┘

transferResp:
  - 通常为空 (表示成功)
  - 或包含校验和等验证信息
```

**处理逻辑**:
```
收到 0x37
    ↓
检查所有数据块是否已接收 (blockNumber 连续性)
    ↓
验证数据完整性 (校验和)
    ↓
编程 Flash (写入模拟存储器)
    ↓
返回 0x77 (成功) 或 0x7F 37 NRC (失败)
```

##### 3.4 请求上传 (0x35 - Request Upload) [可选]

**功能**: 读取已编程的固件数据，用于验证刷写成功或备份现有固件。与0x36配套使用，流程与下载相反。

**请求格式**:
```
┌──────────────┬──────────────┬──────────────┬──────────────┬──────────────┐
│ SID (0x35)   │ dataFormat   │ addressAnd   │ addressAnd   │ addressAnd   │
│ 1 Byte       │ Identifier   │ LengthFormat │ LengthFormat │ LengthFormat │
│              │ 1 Byte       │ 1 Byte       │ 1-4 Bytes    │ 1-4 Bytes    │
└──────────────┴──────────────┴──────────────┴──────────────┴──────────────┘

dataFormatIdentifier: 数据格式 (通常与0x34一致，0x00=二进制)
addressAndLengthFormatIdentifier: 地址和长度格式
```

**示例**:
```
请求: 0x35 0x00 0x44 0x00 0x08 0x00 0x00 0x00 0x80 0x00
    └─ 上传 ─┘ └─ 格式 ─┘ └──── 地址 0x00080000 ────┘ └─ 大小 0x80000 (512KB) ─┘

响应: 0x75 0x00 0x44 [MaxLength_Low] [MaxLength_High]
      └─ 最大块大小 (与下载可能不同)
```

**处理流程**:
```
收到 0x35 请求
    ↓
验证会话与权限 (编程会话 + Level 3)
    ↓
解析地址和大小
    ↓
检查地址范围 (可读区，不包括受保护区)
    ↓
计算块大小和帧数
    ↓
返回 0x75 + maxBlockSize
```

**与0x36的交互**:
- 上传流程与下载对称：0x35→0x36(读回数据)→0x37(退出)
- blockNumber 机制相同，0x01-0xFF循环
- 流量控制：发送方(ECU) 返回数据，接收方(Tester) 返回流控帧

**权限与限制**:
```
可读区域:
  - Public DID 区 (0xF1xx 等) - 任何会话
  - Application 区 - 编程会话 + Level 3
  - Parameter 区 - 编程会话 + Level 3
  - Backup 区 - 编程会话 + Level 3

不可读:
  - Bootloader 区 (返回 NRC 0x31 requestOutOfRange)
  - 诊断元数据区 (可选受保护)
```

---

#### 阶段 4: 验证 (Verification)

**验证方式**:

**执行顺序 (推荐)**:
1) 读取镜像头部并检查 Magic/版本/地址范围。
2) 依据头部的 ImageSize 做整镜像 CRC32 校验,结果写入 `firmware_crc32`。
3) 若启用签名,在 CRC 通过后再验证签名,失败则保持 `app_valid=0`。
4) 校验通过后更新 BootFlags:`app_valid=1`,`active_bank` 切换至已编程分区,`update_pending=0`。

##### 4.1 校验和验证 (Checksum)

**例程: 检查内存完整性 (0x31 0x01 0x01)**

```
请求: 0x31 0x01 0x01 0x44 [Addr] [Size] [Checksum]
      └─ 启动例程 ─┘ └─ 校验和例程 ─┘

响应: 0x71 0x01 0x01 0x00 (校验成功)
     或 0x7F 31 0x35 (校验失败)
```

**校验和算法**:
- **方法1**: 简单累加和 (8-bit)
  ```c
  uint8_t checksum = 0;
  for (int i = 0; i < length; i++) {
      checksum += data[i];
  }
  ```

- **方法2**: CRC-32
  ```c
  uint32_t crc = crc32(0, data, length);
  ```

---

##### 4.2 签名验证 (Signature Verification) - 可选

**前提**: 固件包含数字签名

**验证流程**:
```
1. 读取固件中的签名字段
2. 使用公钥验证签名
3. 验证证书链
4. 检查签名有效期
5. 返回验证结果
```

**安全性**:
- 私钥保护: 私钥仅在签名时使用,不存储在 ECU
- 公钥嵌入: 公钥固化在 Bootloader 中
- 签名算法: RSA-2048 或 ECDSA-P256

---

#### 阶段 5: 后处理 (Post-Programming)

**服务序列**:
```
1. 0x31 0x03 0x02 (检查编程依赖)
   → 0x71 0x03 0x02 0x00 (无依赖)

2. 0x28 0x00 0x01 (使能 RX 和 TX)
   → 0x68 0x00 0x01

3. 0x11 0x01 (ECU 硬复位)
   → 0x51 0x01 (复位确认)
```

**重启后的行为**:
```
ECU 重启
    ↓
Bootloader 检查应用程序
    ↓
验证应用程序有效性标志
    ↓
    ┌─────┴─────┐
有效标志 = 0xAA55   其他值
    │                 │
    ↓                 ↓
跳转到应用程序    等待刷写或进入恢复模式
```

### 4.5 按地址写入内存 (0x3D - Write Memory by Address)

**功能**: 写入小块数据（如参数、标定号、系统标志）到指定内存地址，适用于不需要完整刷写流程的场景。

**典型应用场景**:
- 修改 VIN、零件号、标定数据
- 更新参数区配置
- 激活诊断功能

**请求格式**:
```
┌──────────────┬──────────────┬──────────────┬────────────┬────────────────┐
│ SID (0x3D)   │ addressAnd   │ addressAnd   │ addressAnd │ dataLength     │
│ 1 Byte       │ LengthFormat │ LengthFormat │ LengthFormat│ (根据格式)     │
│              │ 1 Byte       │ 1-4 Bytes    │ 1-4 Bytes  │                │
└──────────────┴──────────────┴──────────────┴────────────┴────────────────┘

addressAndLengthFormatIdentifier:
  - Bit 7-4: 地址字节数
  - Bit 3-0: 长度字节数

dataLength: 要写入的数据长度
```

**示例: 修改 VIN**
```
请求: 0x3D 0x22 [Addr:0x0FB000] [Length:0x11] [17字节 VIN数据]
      └─ 写内存 ─┘ └─ 2B地址,2B长度 ─┘

响应: 0x7D 0x22
      └─ 写内存成功
```

**权限与保护**:
```
参数区 (0x0F0000-0x0F7FFF):
  - 编程会话 + Level 3: 可写
  - Default Session: 仅可读
  - 返回 NRC 0x22 (conditionsNotCorrect) 若权限不足

Bootloader 区 (0x000000-0x00FFFF):
  - 禁止写入，返回 NRC 0x31 (requestOutOfRange)
  - 防止 Bootloader 被覆盖

诊断元数据区:
  - 可配置为只读，返回 NRC 0x22
  - 仅通过 0x27 解锁的操作修改
```

**处理流程**:
```
收到 0x3D 请求
    ↓
验证会话与权限 (编程会话 + Level 3)
    ↓
解析地址、长度和数据
    ↓
检查地址范围 (不能是 Bootloader)
    ↓
检查数据长度 (通常 ≤ 256 字节)
    ↓
验证数据有效性 (范围检查、类型检查)
    ↓
写入内存（原子操作）
    ↓
返回 0x7D (成功) 或 0x7F 3D NRC (失败)
```

**原子性保护**:
```
写入规则:
  - 单次写入要么全成功，要么全失败
  - 若电源中断，使用校验和恢复
  - 不允许部分写入
  
验证机制:
  - 写入后立即读回验证
  - 比对数据，不匹配则返回错误
  - 支持重试（最多 3 次）
```

**与 0x34/0x36 的区别**:

| 特性 | 0x34/0x36 | 0x3D |
|------|-----------|------|
| 用途 | 完整固件刷写 | 小块参数修改 |
| 数据长度 | 最多 4095 B/块 | 最多 256 B |
| 块编号 | 需要 blockNumber | 单次请求 |
| 前置条件 | 需要 0x34 初始化 | 直接调用 |
| 流量控制 | 支持 BS/STmin | 无 |
| 原子性 | 多块聚合 | 单块 |
| 回滚支持 | 有（备份区） | 否（直接修改） |

**NRC 错误码**:

| NRC | 触发条件 |
|-----|---------|
| 0x13 | 格式错误，地址/长度格式不符 |
| 0x22 | 权限不足，参数区不可写 |
| 0x31 | 地址越界或为 Bootloader 区 |
| 0x33 | 安全访问被拒绝 |
| 0x72 | 编程失败（模拟 Flash 故障） |

### 5.3 Raw Binary 格式 (.bin)

**格式**: 纯二进制数据,无元数据

**特点**:
- 无地址信息
- 无校验和
- 需要外部指定加载地址

**加载方式**:
```
方式1: UDS 服务指定地址
  - 0x34 请求中包含目标地址

方式2: 配置文件指定
  - config.json: { "load_address": "0x00100000" }
```

### 5.4 应用镜像头部格式 (推荐)

为支持快速校验、版本管理和回滚,固件文件在入口前增加固定长度头部 (对齐 16 字节)。

| 偏移 | 字段 | 长度 | 描述 |
|------|------|------|------|
| 0x00 | Magic | 4 | 0xAA55AA55, 识别有效镜像 |
| 0x04 | HeaderVersion | 1 | 0x01 |
| 0x05 | ImageType | 1 | 0=App,1=Bootloader(禁用刷写),2=Data |
| 0x06 | Reserved | 2 | 对齐 |
| 0x08 | ImageSize | 4 | 不含头部的镜像长度 (字节) |
| 0x0C | EntryPoint | 4 | 程序入口地址 |
| 0x10 | Version | 4 | 语义化版本编码 (如 0x010203) |
| 0x14 | CRC32 | 4 | 镜像 CRC32 |
| 0x18 | SigOffset | 4 | 签名相对偏移 (可为 0 表示无签名) |
| 0x1C | SigLength | 4 | 签名长度 |
| 0x20 | Reserved | 16 | 预留扩展,总头长 0x30 |

**验证顺序**:
1) 校验 Magic 与 HeaderVersion → 2) 检查 ImageSize 与地址范围 → 3) 计算 CRC32 比对 → 4) 若 SigOffset>0,按公钥验证签名 → 5) 将 Version/EntryPoint 写入 BootFlags 便于回滚选择。

## 6. 安全机制

### 6.1 刷写权限控制

**权限等级**:
| 等级 | 操作 | 安全要求 |
|------|------|----------|
| Level 0 | 读取公开数据 | 无需解锁 |
| Level 1 | 读取诊断数据 | 安全解锁 Level 1 |
| Level 2 | 写入参数 | 安全解锁 Level 2 |
| Level 3 | 刷写固件 | 安全解锁 Level 3 |

**安全访问流程** (0x27):
```
Tester                          ECU
  │                              │
  ├─ 0x27 0x01 (请求种子) ───→  │
  │                              ├─ 生成随机种子
  │ ← 0x67 0x01 [Seed] ─────────┤
  │                              │
  ├─ 计算密钥                    │
  │  Key = f(Seed)               │
  │                              │
  ├─ 0x27 0x02 [Key] ─────────→ │
  │                              ├─ 验证密钥
  │                              │
  │ ← 0x67 0x02 (成功) ─────────┤
  │   或 0x7F 27 35 (失败)       │
```

**密钥生成算法**:

**简化版本** (开发阶段):
```c
uint16_t generate_key(uint16_t seed) {
    return seed ^ 0xABCD;  // XOR 常数
}
```

**安全版本** (生产阶段):
```c
#include <openssl/sha.h>

uint32_t generate_key(uint32_t seed) {
    // 使用 HMAC-SHA256
    const uint8_t secret[] = "SecretKey_ECU_V1";
    uint8_t hash[32];
    uint32_t key;

    HMAC(SHA256, secret, sizeof(secret),
         (uint8_t*)&seed, sizeof(seed),
         hash, NULL);

    memcpy(&key, hash, sizeof(key));
    return key;
}
```

**防暴力破解机制**:
```
尝试次数阈值: 3 次
锁定时间: 5 分钟
记录日志: 所有失败尝试
```

### 6.2 Flash 写保护

**保护区域**:
- **Bootloader 区域**: 0x000000 - 0x007FFF (写保护)
- **写保护控制位**: 只能通过 JTAG/SWD 解除 (模拟)

**写入保护检查**:
```c
int check_write_protection(uint32_t address, uint32_t size) {
    if (address < 0x008000) {
        return ERROR_WRITE_PROTECTED;
    }
    return SUCCESS;
}
```

---

### 6.3 刷写完整性保护

**完整性检查**:
1. **块序号检查**: 确保所有数据块连续且无丢失
   ```c
   if (block_number != expected_block_number) {
       return ERROR_SEQUENCE_ERROR;
   }
   ```

2. **数据校验和**: 每个数据块带校验和
   ```c
   uint8_t checksum = compute_checksum(data, length);
   if (checksum != received_checksum) {
       return ERROR_CHECKSUM;
   }
   ```

3. **整体验证**: 刷写完成后验证整个固件
   ```c
   uint32_t checksum = compute_crc32(firmware_start, firmware_size);
   if (checksum != expected_checksum) {
       return ERROR_VERIFICATION_FAILED;
   }
   ```

### 6.4 刷写特有的 NRC 处理指南

本节补充刷写特定场景下的NRC码，完整NRC表见 [03-UDS_Protocol_Design.md#8.1](03-UDS_Protocol_Design.md#L8.1)。

**刷写流程中的关键 NRC 码**:

| NRC | 服务 | 触发场景 | 恢复策略 |
|-----|------|---------|---------|
| 0x22 | 0x34 | 版本过低，禁止降级 | 检查版本号，使用更高版本重试或联系OEM |
| 0x22 | 0x34 | 刷写已在进行中 (update_pending=1) | 等待超时清理（5分钟）或重启ECU |
| 0x24 | 0x36 | 块序号错误或乱序 | 从正确块号重新发送，若超过3次则重新0x34 |
| 0x31 | 0x34 | 地址越界或为Bootloader区 | 检查刷写地址范围，确保在应用程序区 |
| 0x31 | 0x3D | 试图写入Bootloader或受保护区 | 仅写入参数区(0x0F0000-0x0F7FFF) |
| 0x72 | 0x37 | 校验和验证失败 | 重新读取数据，逐块检查，重新刷写 |
| 0x72 | 0x37 | 签名验证失败 | 检查固件签名，重新签名后刷写 |
| 0x73 | 0x36 | 连续帧序列计数器错误 | ISO-TP 传输层错误，重新建立连接重试 |
| 0x78 | 任意 | 刷写校验需要更多时间 | 等待 P2* 时间后重新查询结果 |

**刷写失败的逐级回退策略**:

```
0x37 返回错误 (校验失败)
    │
    ├─ NRC 0x72 (编程失败)
    │   └─ 策略: 清除本次刷写，保留原应用，诊断仪重新发起刷写
    │
    ├─ NRC 0x24 (序列错误)
    │   └─ 策略: 若备用区已部分刷写，清除；重新0x34启动
    │
    └─ NRC 0x78 (pending)
        └─ 策略: 等待后查询结果，或发起0x31诊断程序查询状态

若连续失败3次:
    ├─ 诊断仪应保存现场日志
    ├─ 提示用户联系技术支持
    └─ ECU 可进入恢复模式，等待下次修复尝试
```

**防止级联失败的设计**:
```
原则1: 隔离新旧数据
  ├─ 正常刷写到备用区 (不覆盖主区)
  └─ 失败时主区保持可用，用户可启动回滚应用

原则2: 幂等重试
  ├─ 重复相同的0x36请求应返回相同的0x76响应
  └─ 支持自动重试，最多3次后放弃

原则3: 断电保护
  ├─ update_pending=1 时若中断，不自动切换启动
  ├─ 重启后进入恢复模式
  └─ 允许重新启动刷写或回滚

原则4: 日志记录
  ├─ 每次失败都记录时间、块号、NRC码
  ├─ 支持故障后的根本原因分析
  └─ 用于改进刷写算法和工具
```

## 7. 异常处理和恢复

### 7.1 异常场景

| 异常 | 原因 | 处理策略 |
|------|------|----------|
| **刷写中断** | 电源中断、CAN 故障 | 保留原数据,标记刷写失败 |
| **校验失败** | 数据损坏 | 重试或中止 |
| **密钥错误** | 非法访问 | 锁定 5 分钟,记录日志 |
| **地址越界** | 非法地址 | 拒绝刷写,返回错误 |
| **Flash 错误** | 模拟 Flash 故障 | 报告错误,停止刷写 |

### 7.2 恢复模式

**触发条件**:
- 应用程序无效标志 ≠ 0xAA55
- 上电检测到刷写未完成标志

**恢复流程**:
```
系统上电
    ↓
Bootloader 检测到 App 无效
    ↓
进入恢复模式
    ↓
等待诊断仪连接
    ↓
重新刷写固件
```

**恢复模式特点**:
- 不跳转到应用程序
- 所有 CAN 报文都响应
- 支持完整的刷写流程

---

### 7.3 刷写失败恢复

**场景**: 刷写过程中断电

**恢复策略**:
```
1. 重新上电
2. Bootloader 检测刷写未完成标志
3. 进入恢复模式
4. 诊断仪重新发送刷写请求
5. Bootloader 擦除不完整数据
6. 重新刷写
```

**双分区回滚**:
- 若 `app_valid=0` 且存在旧版本在另一分区,可切换 `active_bank` 回滚启动,保证车辆可用性。
- 回滚前再次校验旧版本 CRC/签名,避免使用损坏镜像。

**实现**:
```c
// Flash 中存储刷写状态
struct FlashStatus {
    uint32_t magic;            // 0xABCD1234
    uint8_t in_progress;       // 0/1
    uint32_t current_address;  // 当前刷写到哪个地址
    uint32_t expected_size;    // 期望的总大小
};
```

## 8. 性能优化

### 8.1 刷写速度优化

**策略**:
1. **增大块大小**: maxBlockSize 设为 4095 字节 (UDS 上限)
2. **减少 STmin**: 流控帧设置最小帧间隔 (0 = 最小)
3. **并行处理**: 接收数据的同时编程 Flash

**性能指标**:
```
未优化: 5 KB/s
优化后: 15 KB/s
目标: ≥ 10 KB/s
```

---

### 8.2 Flash 编程优化

**批量编程**:
```c
// 模拟: 不需要真实编程时间
// 真实场景: 使用页编程模式
int flash_program_page(uint32_t address, const uint8_t *data, uint16_t size);
```

**后台编程** (可选):
- 使用独立线程进行 Flash 编程
- 主线程继续接收数据

### 8.3 安全与恢复检查清单

- **原子提交**: 仅在校验成功后才切换 `active_bank`, 并写 `app_valid=1`；切换前必须先清除 `update_pending`。
- **回滚保护**: 若目标版本低于当前版本且未设置强制覆盖标志, 则拒绝刷写并返回 NRC 0x31。
- **刷写窗口限制**: 配置最大镜像大小与允许地址段，防止覆盖 Bootloader/元数据区；超出范围立即返回 NRC 0x31。
- **看门狗/电源中断策略**: 支持在编程阶段禁止重启; 如果中断发生, 重启后根据 `update_pending` 进入恢复模式并优先校验备份区。
- **日志与追溯**: 记录刷写请求来源、版本、CRC、写入区域、耗时与结果; 失败场景记录错误码和偏移地址。
- **接口幂等性**: 重复的 `0x36` 块号请求若数据一致应返回肯定响应, 不一致则返回 NRC 0x24 并终止当前刷写会话。
