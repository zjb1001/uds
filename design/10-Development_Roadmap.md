## 1. 项目规划概述

### 1.1 开发阶段划分

```
┌─────────────────────────────────────────────────────────┐
│                    项目时间线                             │
├─────────────────────────────────────────────────────────┤
│  Phase 1    Phase 2    Phase 3    Phase 4    Phase 5    │
│  基础框架    核心服务    刷写功能    测试优化    发布维护  │
```

### 1.2 里程碑

| 里程碑 | 目标日期 | 交付物 |
|--------|----------|--------|
| **M1: 框架完成** | 第 4 周 | SocketCAN 封装,ISO-TP 传输层,错误/超时框架骨架 |
| **M2: 核心功能** | 第 8 周 | 核心服务实现(含 NRC/0x78),多 ECU 会话隔离,可配置 P2/P2*/S3 |
| **M3: 刷写支持** | 第 12 周 | Bootloader/Flash 模拟,刷写流程,Python FlashTool 原型,API 冻结点 |
| **M4: 测试完成** | 第 16 周 | 单元测试/集成测试/多 ECU 并发/性能达标(含安全锁定回归) |
| **M5: 正式发布** | 第 20 周 | 文档完善,v1.0.0 发布 |

## 11. 风险与依赖

| 风险/依赖 | 影响 | 缓解措施 | 触发指标 |
|-----------|------|----------|----------|
| SocketCAN/vCAN 兼容性 (不同内核版本) | 阻塞集成/测试 | 维护已验证内核列表; 提供容器化环境 | 新环境驱动缺失或报错 |
| Rust/Python 工具链版本漂移 | 构建失败/行为差异 | 锁定工具链版本, CI 使用固定镜像; 定期升级验证 | 构建时间 >15% 或警告增加 |
| 多 ECU 并发隔离复杂度 (Phase 2-4 的核心难点) | 里程碑 M2-M4 延误; 数据竞争/锁定问题 | 提前完成 ServiceRegistry/DiagnosticSession 等核心接口设计(第08章); 并发测试框架与压力测试提前(周 13 前); 代码评审重点关注 ecu_id 隔离和配置超时线程安全性 | 集成测试出现非预期的会话/安全状态泄露或超时配置冲突; PR 中涉及多 ECU 的评审 >7 天 |
| 性能未达标 (刷写速率/延迟) | 里程碑 M3/M4 延误 | 预留性能优化冲刺; 提前建立基准与回归阈值 | 基准退化 >10% |
| 安全算法/依赖漏洞 | 发布阻塞 | 启用 `cmake audit`、`pip-audit`; 建立紧急修复流程 | 高危 CVE 检出 |
| 关键人风险 | 交付延迟 | 模块代码所有权和评审轮值; 关键模块双人备份 | 核心 PR >5 天未评审 |

## 12. 里程碑验收门槛

- **M1**: vCAN 回环/路由测试通过; uds-can/uds-tp 单元覆盖率 ≥80%; 传输层超时/错误分类可观测。
- **M2**: 核心服务 (0x10/0x3E/0x27/0x22/0x2E/0x19) 集成测试全通过; 关键 NRC(含 0x78) 行为符合协议; 多 ECU 会话/安全状态隔离验证通过; P2/P2*/S3 可配置并有测试。
- **M3**: 刷写端到端成功率 ≥99.5%; 恢复/回滚测试通过; 性能达到 10 KB/s; 公开 API 冻结(对齐第08章接口与第09章测试): Rust/Python 外部可见接口、错误枚举与关键行为在 v0.x 阶段保持兼容,破坏性变更需显式版本升级与迁移说明。
- **M4**: 覆盖率 ≥85%; 压测/多 ECU 并发场景通过; 性能与安全基线稳定(含安全锁定回归)。
- **M5**: 文档齐全 (设计/用户/API); 发布说明与已知问题列表发布; CI 全绿。
